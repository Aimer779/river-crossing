# PRD：《郝哥求生指南》

版本：v1.0
产品类型：Web 端 2D 规则解谜小游戏
推荐技术栈：Phaser 3 + TypeScript + Vite
目标平台：桌面浏览器、移动端横屏浏览器
玩法原型：Missionaries and Cannibals 魔改版
核心角色：华强、郝哥
核心目标：在不让郝哥被华强“拿捏”的情况下，把所有人送到右岸。

---

# 1. 项目概述

《郝哥求生指南》是一款单关卡逻辑解谜小游戏。

玩家需要操作 3 个华强和 3 个郝哥乘船过河。船每次最多坐 2 人，至少 1 人才能开船。玩家需要安排合理的往返顺序，把所有角色从左岸运送到右岸。

游戏的核心限制是：任意一岸只要出现“华强数量大于郝哥数量”，并且该岸还有郝哥，郝哥就会被华强拿捏，游戏失败。

游戏整体风格建议为“Q 版市井荒诞风”。玩法本身保持经典逻辑谜题的严谨性，表现层可以做得更夸张、更有梗。

---

# 2. 产品目标

本项目的目标不是做复杂系统，而是把一个经典逻辑谜题包装成一个完整、有趣、可玩的整活小游戏。

核心目标如下：

第一，玩家能快速理解规则，不需要长时间阅读说明。

第二，所有规则判断必须稳定，不能因为连点、撤销、动画中断、AI 演示等情况导致状态错乱。

第三，游戏需要有足够明确的反馈。玩家非法操作时知道为什么错，失败时知道哪一岸出了问题。

第四，整活表达要围绕玩法服务，不能让动画、语音、梗影响游戏可读性。

第五，底层规则模块需要和 Phaser 表现层解耦，后续可以方便扩展更多角色数量、船容量、关卡或主题皮肤。

---

# 3. 核心玩法

## 3.1 角色

游戏中有两类角色。

华强：规则中的强势方。
郝哥：规则中的脆弱方。

初始状态：

左岸：3 个华强 + 3 个郝哥
右岸：0 个华强 + 0 个郝哥
船：停在左岸，船上为空

玩家目标：

把所有华强和郝哥全部送到右岸。

---

## 3.2 船规则

船每次最多坐 2 人。
船每次至少坐 1 人才能开船。
船只能在左岸和右岸之间往返。
船每次开到对岸后，船上角色自动下船。
MVP 版本不做“到岸后手动下船”。

船允许的载客组合只有 5 种：

1 个华强
1 个郝哥
2 个华强
2 个郝哥
1 个华强 + 1 个郝哥

---

## 3.3 失败规则

任意一岸同时满足以下两个条件，则游戏失败：

该岸郝哥数量大于 0；
该岸华强数量大于郝哥数量。

规则公式：

```ts
bank.haoge > 0 && bank.huaqiang > bank.haoge
```

示例：

左岸 3 华强 + 1 郝哥：失败。
右岸 2 华强 + 1 郝哥：失败。
左岸 3 华强 + 0 郝哥：不失败，因为这一岸没有郝哥。
右岸 0 华强 + 2 郝哥：不失败。

失败主题表达：

“郝哥被拿捏了。”
“这一岸华强太多，郝哥顶不住了。”
“局势失控。”

---

## 3.4 胜利规则

当所有角色都到达右岸，并且船也在右岸时，游戏胜利。

胜利状态：

左岸：0 华强 + 0 郝哥
右岸：3 华强 + 3 郝哥
船：右岸

胜利主题表达：

“郝哥安全上岸。”
“全员成功渡河，局势稳住了。”
“本局共用 X 步。”

---

# 4. 关键产品决策

## 4.1 是否自动下船

MVP 版本采用自动下船。

完整一步定义为：

玩家选择 1 到 2 个当前岸角色上船；
玩家点击开船；
船移动到对岸；
船上角色自动下船；
系统检查胜负；
记录一步历史。

不建议 MVP 做手动下船。原因是手动下船会引入中间状态：船靠岸后，船上的角色到底算不算该岸人数？这会让规则判断、动画、提示、撤销都变复杂。自动下船可以保证“一次开船就是一步”，状态更干净。

---

## 4.2 是否允许失败操作

本 PRD 建议：允许玩家执行会导致失败的合法移动。

也就是说：

空船、超载、点错岸，这些属于非法操作，直接阻止。
但如果船上人数合法，只是开过去以后会导致某岸“华强多于郝哥”，则允许船开过去，然后触发失败演出。

这样更适合整活游戏，因为失败本身就是表现机会，可以播放华强压迫感动画、郝哥慌张动画、镜头震动和失败文案。

---

## 4.3 是否做撤销

MVP 建议必须做撤销。

原因：

撤销成本低。
能明显降低挫败感。
玩家可以自由试错。
也方便开发调试。

失败后也建议允许玩家点击“撤销一步”，回到失败前的状态，而不是只能重开。

---

# 5. 游戏流程

## 5.1 主流程

玩家进入标题页。
点击开始游戏。
进入主游戏场景。
玩家点击当前岸角色，让角色上船。
玩家点击开船。
船移动到对岸。
角色自动下船。
系统检查是否胜利或失败。
如果未结束，继续下一步。
如果胜利，进入胜利结算。
如果失败，播放失败演出后进入失败结算。
玩家可选择重开、撤销或返回标题。

---

## 5.2 单步操作流程

一次完整移动流程如下：

1. 玩家选择角色上船。
2. 系统检查船容量。
3. 玩家点击开船。
4. 系统检查船上是否有人。
5. 锁定输入。
6. 播放开船动画。
7. 更新船的位置。
8. 船上角色自动下船。
9. 更新左右岸人数。
10. 步数 +1。
11. 写入历史记录。
12. 检查失败。
13. 检查胜利。
14. 解锁输入或进入结算。

# 7. 状态模型

规则逻辑必须独立于 Phaser。建议放在：

`src/core/rules.ts`

推荐类型：

```ts
type Side = 'left' | 'right';

type BankState = {
  huaqiang: number;
  haoge: number;
};

type GameStatus = 'playing' | 'win' | 'lose';

type GameState = {
  left: BankState;
  right: BankState;
  boatSide: Side;
  boat: BankState;
  status: GameStatus;
  steps: number;
};
```

移动结构：

```ts
type Move = {
  huaqiang: number;
  haoge: number;
  from: Side;
  to: Side;
};
```

初始状态：

```ts
const initialState: GameState = {
  left: {
    huaqiang: 3,
    haoge: 3
  },
  right: {
    huaqiang: 0,
    haoge: 0
  },
  boatSide: 'left',
  boat: {
    huaqiang: 0,
    haoge: 0
  },
  status: 'playing',
  steps: 0
};
```

# 8. 核心规则函数

## 8.1 判断某岸是否安全

```ts
function isBankSafe(bank: BankState): boolean
```

逻辑：

如果郝哥数量为 0，则安全。
如果华强数量小于等于郝哥数量，则安全。
否则不安全。

---

## 8.2 判断整个状态是否安全

```ts
function isStateSafe(state: GameState): boolean
```

逻辑：

左岸安全且右岸安全，则状态安全。
任意一岸不安全，则状态不安全。

---

## 8.3 判断是否胜利

```ts
function isWin(state: GameState): boolean
```

胜利条件：

左岸华强 = 0
左岸郝哥 = 0
右岸华强 = 3
右岸郝哥 = 3
船在右岸

---

## 8.4 判断是否失败

```ts
function isLose(state: GameState): boolean
```

失败条件：

左岸不安全，或右岸不安全。

---

## 8.5 获取当前可执行移动

```ts
function getPossibleMoves(state: GameState): Move[]
```

返回船当前所在岸理论上可以执行的移动，不判断移动后是否失败。

可能组合：

```ts
[
  { huaqiang: 1, haoge: 0 },
  { huaqiang: 0, haoge: 1 },
  { huaqiang: 2, haoge: 0 },
  { huaqiang: 0, haoge: 2 },
  { huaqiang: 1, haoge: 1 }
]
```

需要过滤：

船当前岸人数不足的组合。
空组合。
超过船容量的组合。

---

## 8.6 获取安全移动

```ts
function getSafeMoves(state: GameState): Move[]
```

返回移动后不会失败的移动。

用途：

提示系统
AI 自动演示
BFS 求解器

---

## 8.7 应用移动

```ts
function applyMove(state: GameState, move: Move): ApplyMoveResult
```

推荐返回结构：

```ts
type ApplyMoveResult = {
  ok: boolean;
  state?: GameState;
  reason?: IllegalReason;
};
```

非法原因枚举：

```ts
type IllegalReason =
  | 'EMPTY_BOAT'
  | 'OVER_CAPACITY'
  | 'NOT_ENOUGH_PEOPLE'
  | 'WRONG_SIDE'
  | 'GAME_NOT_PLAYING'
  | 'BOAT_MOVING';
```

注意：

`applyMove` 不应该直接修改原始 state。
必须返回新对象。
总人数必须守恒。
船移动后，船上人数清零。
船位置切换到对岸。
steps +1。

---

## 8.8 BFS 求解器

```ts
function solveByBFS(state: GameState): Move[]
```

要求：

从当前状态开始搜索最短胜利路径。
只使用安全移动。
返回一组 Move。
如果无解，返回空数组。
不允许硬编码答案。

用途：

提示按钮
AI 自动演示
自动化测试
确认规则可解

# 10. 主交互需求

## 10.1 点击岸上角色上船

条件：

角色必须在船当前所在岸。
船上人数少于 2。
游戏状态为 playing。
船没有正在移动。
AI 演示没有运行。

行为：

角色进入船上选择状态。
角色移动到船上空座位。
船上座位显示更新。
开船按钮状态更新。

非法提示：

船不在这一岸。
船最多只能坐 2 个人。
船正在动，先别急。
现在不能操作。

---

## 10.2 点击船上角色取消上船

条件：

角色已经在船上。
船尚未开动。
游戏状态为 playing。

行为：

角色回到当前岸队列。
释放船上座位。
开船按钮状态更新。

---

## 10.3 点击开船

条件：

船上至少 1 人。
船上最多 2 人。
游戏状态为 playing。
船没有移动中。
AI 演示没有运行。

行为：

锁定输入。
播放开船动画。
船移动到对岸。
角色自动下船。
更新状态。
记录历史。
步数 +1。
检查失败。
检查胜利。
进入下一状态。

非法提示：

船上没人，开不了船。
这船最多坐 2 个人。
船正在动，先别急。
现在不能开船。

失败处理：

如果移动后的状态触发失败，播放失败演出，然后进入失败结算。

---

## 10.4 撤销

行为：

恢复到上一步完整状态。
同步恢复船位置、角色位置、人数、步数、历史记录。
清除提示高亮。
如果当前处于失败状态，也允许撤销回到失败前一步。

限制：

没有历史状态时，撤销按钮置灰。
船移动过程中不能撤销。
AI 演示过程中不能撤销，除非先停止演示。

提示：

已经没有上一步了。

---

## 10.5 提示

行为：

调用 BFS。
从当前状态获取最短路径。
取第一步作为推荐。
高亮推荐角色。
显示提示文案。

提示文案示例：

试试让 2 个华强上船。
试试让 1 个华强回来。
试试让 2 个郝哥上船。
试试让 1 个华强和 1 个郝哥回来。

如果当前状态无解：

当前局面无解，建议撤销一步。

要求：

提示必须基于当前状态实时计算。
不允许写死提示。
撤销后再次点击提示，必须重新计算。

---

## 10.6 AI 自动演示

行为：

点击“AI 演示”后，进入自动模式。
系统调用 BFS 获取最短安全路径。
依次执行每一步：

自动选择角色。
播放上船动画。
点击开船逻辑自动触发。
船移动。
角色自动下船。
等待 0.8-1.2 秒。
执行下一步。

限制：

AI 演示期间禁用玩家点击。
提供“停止演示”按钮。
停止后保留当前状态。
停止后玩家可以继续手动操作。
AI 演示过程中不能撤销，除非先停止演示。

# 12. 历史记录

主界面显示最近若干步操作。

格式：

第 1 步：2 个华强 → 右岸
第 2 步：1 个华强 ← 左岸
第 3 步：2 个华强 → 右岸
第 4 步：1 个华强 ← 左岸
第 5 步：2 个郝哥 → 右岸

要求：

历史记录与状态栈一致。

# 25. 实现重点

这个项目最重要的不是 Phaser 动画，而是规则内核。

请优先完成：

```ts
isBankSafe()
isStateSafe()
isWin()
isLose()
getPossibleMoves()
getSafeMoves()
applyMove()
solveByBFS()
```

所有游戏表现都围绕这些函数展开。

不要把规则散落写在 GameScene 里。
不要让动画直接决定规则状态。
不要硬编码最优解。
不要让提示系统和 AI 演示各写一套逻辑。
不要让 UI 显示的数据和真实 GameState 分离。

正确结构是：

规则内核负责判断。
GameScene 负责调用。
动画系统负责表现。
UI 负责显示。
BFS 负责提示和自动演示。
历史栈负责撤销。

一句话实现原则：

**先做一个完全正确的规则游戏，再把它包装成一个好笑的整活游戏。**
