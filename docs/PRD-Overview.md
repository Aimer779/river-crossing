# PRD：《郝哥求生指南》

版本：v1.0
产品类型：Web 端 2D 规则解谜小游戏
推荐技术栈：Phaser 3 + TypeScript + Vite
目标平台：桌面浏览器、移动端横屏浏览器
玩法原型：Missionaries and Cannibals 魔改版
核心角色：华强、郝哥
核心目标：在不让郝哥被华强"拿捏"的情况下，把所有人送到右岸。

---

## 1. 项目概述

《郝哥求生指南》是一款单关卡逻辑解谜小游戏。

玩家操作 3 个华强和 3 个郝哥乘船过河。船每次最多坐 2 人，至少 1 人才能开船。玩家需安排合理往返顺序，把所有角色从左岸运送到右岸。

核心限制：任意一岸只要出现"华强数量 > 郝哥数量"，且该岸还有郝哥，郝哥就会被华强拿捏，游戏失败。

游戏整体风格为"Q 版市井荒诞风"。玩法保持经典逻辑谜题严谨性，表现层可更夸张、更有梗。

---

## 2. 产品目标

本项目目标是把经典逻辑谜题包装成完整、有趣、可玩的整活小游戏。

- 玩家能快速理解规则，无需长时间阅读说明。
- 所有规则判断必须稳定，不因连点、撤销、动画中断、AI 演示等导致状态错乱。
- 游戏需有足够明确的反馈。玩家非法操作时知道为什么错，失败时知道哪一岸出了问题。
- 整活表达围绕玩法服务，动画/语音/梗不影响游戏可读性。
- 底层规则模块与 Phaser 表现层解耦，方便后续扩展更多角色数量、船容量、关卡或主题皮肤。

---

## 3. 核心玩法

### 3.1 角色

- 华强：规则中的强势方。
- 郝哥：规则中的脆弱方。

初始状态：左岸 3 华强 + 3 郝哥；右岸 0 华强 + 0 郝哥；船停左岸，船上为空。

玩家目标：把所有角色送到右岸。

### 3.2 船规则

- 船每次最多坐 2 人，至少坐 1 人才能开船。
- 船只能在左岸和右岸之间往返。
- 船每次开到对岸后，船上角色自动下船。MVP 不做"到岸后手动下船"。

允许载客组合（5 种）：1 华强 / 1 郝哥 / 2 华强 / 2 郝哥 / 1 华强 + 1 郝哥。

### 3.3 失败规则

任意一岸同时满足以下两条件则游戏失败：
- 该岸郝哥数量 > 0
- 该岸华强数量 > 郝哥数量

规则公式：`bank.haoge > 0 && bank.huaqiang > bank.haoge`

示例：左岸 3 华强 + 1 郝哥 → 失败；右岸 2 华强 + 1 郝哥 → 失败；左岸 3 华强 + 0 郝哥 → 不失败（无郝哥）；右岸 0 华强 + 2 郝哥 → 不失败。

失败主题表达："郝哥被拿捏了。" / "这一岸华强太多，郝哥顶不住了。" / "局势失控。"

### 3.4 胜利规则

所有角色都到达右岸，且船也在右岸时，游戏胜利。

胜利状态：左岸 0 华强 + 0 郝哥；右岸 3 华强 + 3 郝哥；船在右岸。

胜利主题表达："郝哥安全上岸。" / "全员成功渡河，局势稳住了。" / "本局共用 X 步。"

---

## 4. 关键产品决策

### 4.1 是否自动下船

MVP 采用自动下船。完整一步定义为：玩家选择 1~2 个当前岸角色上船 → 点击开船 → 船移动到对岸 → 角色自动下船 → 系统检查胜负 → 记录一步历史。

不建议 MVP 做手动下船，会引入中间状态（船靠岸后船上角色是否算该岸人数），让规则判断、动画、提示、撤销都变复杂。自动下船保证"一次开船就是一步"，状态更干净。

### 4.2 是否允许失败操作

允许玩家执行会导致失败的合法移动。空船、超载、点错岸属于非法操作，直接阻止；但船上人数合法、只是开过去会导致某岸"华强多于郝哥"，则允许开过去，然后触发失败演出。这样更适合整活游戏，失败本身是表现机会。

### 4.3 是否做撤销

MVP 必须做撤销。原因：成本低、降低挫败感、玩家可自由试错、方便开发调试。失败后也允许点击"撤销一步"回到失败前状态，而不是只能重开。

## 5. 优先级

### P0 必须完成
核心规则、状态机、开船、自动下船、失败判断、胜利判断、撤销、提示、重新开始、基础 UI、基础动画、静态部署。

### P1 应该完成
AI 自动演示、历史记录、正式美术、音效、角色语音、移动端横屏适配、胜利/失败演出。

### P2 可选完成
更多整活语音、粒子效果、多关卡、限步挑战、计时挑战、分享截图、排行榜、皮肤切换。
